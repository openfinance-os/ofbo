Adapting the Build Harness to AI-DLC Tools

The OFBO harness was built so the AI authoring tool is swappable. Regulated control lives in deterministic CI gates, git-trailer provenance, and branch protection — not in any one editor. This page shows the adaptation contract every AI-driven-development-lifecycle tool must satisfy, classifies the tool landscape by type, and works two examples in full: Cursor and Claude Code. Others slot into the same frame.

The core principle

The methodology is the harness; the tool is the worker

AI-DLC is a tool-agnostic methodology — AWS's own framework states it "works with any IDE, agent, or model." OFBO encodes a stricter instance of it. Because the load-bearing controls are mechanical and pipeline-side, a different tool changes ergonomics, not assurance.

Two tiers of enforcement — only one is tool-specific.

Advisory  The PreToolUse tripwires (PII / spec / test) and worktree isolation are Claude Code primitives. A different tool won't fire them — so this convenience layer must be re-created as tool-neutral git hooks.

Merge-blocking External  The deterministic CI gates (Q1b test-integrity, Q4.5 lineage), the sealed evidence bundle, and human four-eyes via branch protection are tool-independent by design (ADR 0019). They care about the commit, not the editor that wrote it. This is the real safety net, and it survives any tool swap.

Know the type before you score the tool

Four types of AI-DLC tool

The decisive question for a UAE bank is where inference runs and what leaves the perimeter. That is a property of the tool's type, not its brand. Place a new tool in a type first; the egress and outsourcing posture mostly follows.

What actually decides it

The two binding gates — and two common myths

For a UAE Open Finance bank, the regulatory weight falls on exactly two gates. Two things people expect to be blockers are not.

Gate 1 · Data egress (P6, no direct egress)

All external traffic must route through the enterprise egress gateway; code must not leave the bank's boundary uncontrolled. A tool that mandatorily proxies context through its own cloud (Type A) cannot honour this. A tool whose inference runs in the bank's own tenancy (Type B in-region, or Type C air-gapped) can.

Gate 2 · CBUAE outsourcing & audit access

Using an external AI SaaS is outsourcing under CBUAE Circular 14/2021. If material: board approval, non-objection before signing, and contract clauses granting CBUAE on-site/audit access at the provider. The Feb-2026 AI Guidance Note adds: responsibility cannot be outsourced; audit rights, cyber guarantees, immediate-cessation. Easiest to clear when the counterparty is a cloud the bank has already cleared.

Myth 1 · "The code must be UAE-resident"

Residency rules attach to Consumer / transaction / Personal / Payment data — not source code. PII-free code is, on the face of the texts, out of scope. OFBO's synthetic-only / zero-PII mandate keeps it that way. The residency objection is about data, which never enters the tool's context.

Myth 2 · "AI-written code triggers the EU AI Act"

EU AI Act Art. 12 / 17 bind providers of high-risk AI systems and govern the deployed system's runtime logging — not the act of writing code with an assistant (limited-risk at most). What's regulated is what the bank ships, regardless of which tool helped write it. Provenance is a defensible self-imposed control, not a tool mandate.

Adding a new tool to the frame

Onboarding a candidate (GitHub Copilot, Tabnine, Windsurf, Sourcegraph Cody, Devin, OpenHands…) is a fixed worksheet. The page is data-driven, so the last step is a one-line edit.

  1. Classify the type. A cloud-SaaS copilot, B pluggable-backend agent, C self-hosted/air-gapped, or D autonomous cloud agent. This fixes the egress and outsourcing posture before any deep review.
  2. Score Gate 1 (egress). Where does inference physically run? Can it be pinned to the bank's tenancy / an in-region endpoint / the P6 gateway? If it mandatorily proxies through the vendor's cloud, it is blocked for regulated repos.
  3. Score Gate 2 (outsourcing). Who is the data-plane counterparty, and is there an existing CBUAE-cleared relationship to ride? Can the contract grant CBUAE on-site/audit access, immediate cessation, and no-training/ZDR?
  4. Map the harness controls. Mark each row of the matrix native / bridge / blocker. Tripwires and worktree isolation are usually bridge for non-Claude tools (re-implement as git pre-commit / prepare-commit-msg hooks). CI gates and four-eyes are always external.
  5. Wire provenance. Ensure commits carry Co-Authored-By + a Build-Model / session trailer so the ADR 0019 evidence bundle attributes the tool. Editor-side AI tracking is a hint; the CI bundle is the system of record.
  6. Add one data object. Append a record to the TOOLS array in this file's script (name, type, verdict, per-control status). The card and the matrix column generate themselves.
Decision rule of thumb. If the no-direct-egress rule is taken literally for all code: Type B (in-tenancy) and Type C (air-gapped) are eligible; Type A is restricted to non-regulated/synthetic work under an Enterprise DPA, or excluded. Everything downstream of egress — gates, provenance, four-eyes — is tool-neutral and already solved by the harness.
