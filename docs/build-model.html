The ideaWe built a way to build

Open Finance is the example. The model is the deliverable.

Most delivery programmes produce one thing: the thing. This one produced two — the Open Finance Back Office, and the method that built it. The method is the asset that lasts. It takes an ambiguous idea, finds the problem worth solving, and turns it into live, audit-ready software with the controls built in — and it does so the same way every time.

Open Finance is simply the first domain we ran it through. Payments, lending, onboarding, servicing, risk — each can reuse the same machine instead of reinventing one. The point for leadership is not "we shipped a back office." It is "we now have a repeatable, controlled way to ship anything regulated."

The modelA double diamond — discover the right problem, then deliver it

The model is a double diamond. The first diamond is about the problem; the second is about the solution. Each opens up to explore, then narrows to decide. They meet at one artifact — the agreed problem statement, or PRD — the hand-off from discovery to delivery.

Discovery — finding the right problem

Before a line of code, the discovery harness turns a challenge into evidence and an agreed problem. It diverges to gather evidence widely — users, market, regulation, data, current state — then converges on a framed problem, a business case, and a scope, with a funded go/no-go and an optional RFP when it is a build-vs-buy call. Crucially, a discovery is allowed to fail: stopping the wrong problem early is a win, not a waste.

Delivery — shipping it under control

The delivery harness turns the PRD into working software on rails. Every change is specification-first, four-eyes approved, and auto-deployed to a live demo; nothing ships without clearing build, security, integration and data-lineage gates; and no real customer data ever touches the system. Compliance and auditability are properties of the line — not a scramble before an audit.

The brainContext is king — the regulated-entity brain

A regulated entity is not generic, so the model can't be either. What makes it this institution's way of building is context — how it works, how it words things, how it abbreviates, how it is regulated. We treat that as a first-class asset: a regulated-entity brain the harnesses read from at every step and write back to after every cycle.

It has two layers — general practice that ships with the model, and the entity-specific context built up over time. It is seeded from what the institution already has, lives in one governed place rather than in people's heads, and it compounds: every cycle makes it richer, so the next solution is faster and safer.

This is the moat. The model is generic by design and therefore copyable; the trained brain is not. It is where general best practice and the institution's own identity meet — turning "a way to build" into this regulated entity's way of building.

Going forwardHow we build from here

Treat this as the template, not the exception. The discovery harness gives leadership a disciplined way to fund the right problems and stop the wrong ones early. The delivery harness gives teams a factory where control is automatic. The brain makes both improve with every use.

Because every external system is a port and every control is a gate, the same machine adapts to a new product, a new domain, a new regulator or a new vendor. Scaling means adding work to the line — not rebuilding it.
